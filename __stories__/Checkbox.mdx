Checkbox is the canonical control for boolean input. Use it for opt-ins,
multi-select lists, and any standalone toggle whose state can be expressed as
on / off (or, with `indeterminate`, the third "mixed" state used in tree-style
selections). For settings that take effect immediately prefer `Toggle`; for
mutually exclusive choices prefer `Radio` / `RadioGroup`.

        <h2>Anatomy</h2>

        <p>A Checkbox is composed of:</p>

          <li>a <strong>label</strong> — names the option; required for every checkbox, clickable</li>

          <li><strong>Multi-select lists</strong> — independently togglable options where any combination is valid.</li>

        <h3>When <em>not</em> to use a Checkbox</h3>

          <li>Keep error messages actionable — say what to fix, not just what&rsquo;s wrong.</li>

        <h2>Accessible name</h2>

        <p>The visible label is the accessible name (wired up via <code>htmlFor</code>). Never omit the label. If a design legitimately calls for a label-less checkbox (rare; e.g. a row-select cell in a table) provide <code>aria-label</code> describing what the checkbox controls.</p>

        <h2>State announcements</h2>

        <p>Screen readers announce &ldquo;checked&rdquo;, &ldquo;not checked&rdquo;, or &ldquo;mixed&rdquo; when focus lands on the checkbox and when the state changes. The label should describe the <strong>option</strong>, not the state.</p>

        <h2>Required fields</h2>

        <p>Mark required checkboxes with <code>required</code>. The label renders an asterisk and the input gets <code>aria-required</code>. Avoid marking optional checkboxes — clutter scales poorly.</p>

        <h2>Errors</h2>

        <p>When <code>error</code> is set the input is <code>aria-invalid</code> and the message is linked via <code>aria-errormessage</code>. Screen readers announce the error on focus. Keep error text actionable.</p>

        <h2>Disabled checkboxes</h2>

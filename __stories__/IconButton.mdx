IconButton is an icon-only action control. Use it in toolbars, table rows, and
anywhere a labelled `Button` would crowd the layout. Because there is no visible
label, accessibility relies on `ariaLabel` and a hover `tooltip`.

        <h2>Anatomy</h2>

        <p>An IconButton is composed of:</p>

        <ul>
          <li>an <strong>icon</strong> — a glyph that conveys the action</li>
          <li>an <strong>accessible name</strong> — supplied via <code>ariaLabel</code> (or <code>ariaLabelledBy</code>)</li>
          <li>an optional <strong>tooltip</strong> — strongly recommended; doubles as a visible label on hover</li>
          <li>an optional <strong>loader</strong> — replaces the icon while an async action is in progress</li>

        <h2>When to use</h2>

        <ul>
          <li>Toolbars, table row actions, and other dense layouts where a labelled <code>Button</code> would not fit.</li>
          <li>Universally recognised icons (close, settings, more options, search). Avoid bespoke glyphs.</li>
          <li>Pair with a tooltip every time. If the action would be ambiguous without a tooltip, use a labelled <code>Button</code> instead.</li>
        </ul>

        <h2>Variant choice</h2>

        <ul>
          <li><strong>ghost</strong> — the most common choice. Use in toolbars, list rows, dialog headers.</li>
          <li><strong>outlined</strong> — when the button needs to read as standalone (e.g. pagination chevrons).</li>
          <li><strong>contained</strong> — for high-emphasis primary actions (rare; usually paired with a label).</li>
          <li><strong>color="danger"</strong> — for destructive actions inside a confirmed context. Always pair with a tooltip.</li>

        <h2>Icon choice</h2>

        <p>An IconButton is a real <code>{'<button>'}</code> element. The accessibility burden is higher than a labelled <code>Button</code> because there is no visible text — get the name and tooltip right.</p>

        <h2>Accessible name</h2>

        <p>Every IconButton must have an accessible name. Provide one of:</p>

        <ul>
          <li><code>ariaLabel</code> — a short verb phrase describing the action (<code>"Delete row"</code>, not <code>"Trash"</code>).</li>
          <li><code>ariaLabelledBy</code> — when an adjacent visible element already names the button.</li>
          <li><code>ariaHidden</code> — only when the button is decorative or a duplicate of an action elsewhere.</li>
        </ul>

        <p>The TypeScript types enforce that exactly one of these is set.</p>

        <h2>Tooltips</h2>

        <p>A tooltip is not the same as an accessible name. Always provide <code>ariaLabel</code> in addition to <code>tooltip</code>. The tooltip helps sighted users discover the action; the <code>ariaLabel</code> is what assistive tech announces.</p>

        <p>Match the two — they should say the same thing. Conflicting tooltip and aria-label confuse speech-input users.</p>

        <h2>Color is not a meaning</h2>

        <p>Variant colors are invisible to screen readers and unreliable for users with color-vision differences. The <strong>label</strong> must carry the meaning. Don't rely on a red icon to communicate destruction.</p>

        <h2>Target size</h2>

        <p>The regular variant (36×36) and small variant (32×32) both meet the WCAG 2.5.5 minimum target size of 24×24 CSS pixels. Don't shrink the button further with custom CSS.</p>

        <h2>Keyboard</h2>

        <ul>
          <li><code>Tab</code> moves focus to the button.</li>
          <li><code>Enter</code> and <code>Space</code> activate it.</li>
          <li>A visible focus ring is shown on <code>:focus-visible</code>. <strong>Don't override it.</strong></li>
        </ul>

        <h2>Disabled buttons</h2>

        <p>A disabled icon button has no visible label and no clue why it's disabled. Always provide <code>disabledTooltip</code>. If the user needs to interact to learn why it can't be used, prefer keeping the button enabled and showing an error on click.</p>

        <h2>Loading state</h2>

        <p>While <code>loading</code>, the button stays in the DOM, remains focusable, and sets <code>aria-disabled="true"</code>. Pair with an <code>aria-live="polite"</code> region elsewhere announcing progress.</p>
